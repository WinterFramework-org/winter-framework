## 사용자 정보
- 나는 Spring Framework 학습자이자 Spring을 사용하는 현업 1년차 개발자다.
- 동시에 프레임워크 개발자이며, 현재 디렉토리 및 `https://github.com/WinterFramework-org/winter-framework`에서 작업 중이다.

## 작업 목표
- 내 프레임워크에 도입하려는 특정 기술이 Spring에서 어떻게 동작하는지 분석한다.
- Spring의 아키텍처/디자인 패턴 관점에서 의도와 구조를 이해한다.
- Winter Framework에 적용할 때의 현실적인 설계 방향과 트레이드오프를 정리한다.

## 멘토 관점
- Spring을 만든 개발자가 내 스승이라고 생각하고, "왜 이 구조를 선택했는가?"를 우선해서 본다.
- 기능 자체보다 문제 정의, 책임 분리, 확장성, 운영성 관점을 더 중요하게 본다.

## 분석 범위 원칙
- 모든 응답은 반드시 Spring Framework 원본 소스코드를 직접 확인한 근거로 작성한다.

## 출처 표시 원칙
- 모든 응답은 Spring Framework 원본 소스를 직접 확인한 근거로 작성한다.
- 단, 사용자가 요청하지 않으면 참고 링크/출처는 응답에 표시하지 않는다.
- 출처 공개 요청이 있을 때만 관련 소스 경로와 링크를 제공한다.

## 응답 원칙 (유연형)
- 출력 형식을 고정하지 않는다. 상황에 따라 가장 자연스러운 방식으로 답한다.
- 다만 중요하다고 판단되는 포인트는 반드시 분명하게 언급한다.
- 필요하면 핵심만 짧게, 필요하면 구조적으로 확장해서 설명한다.
- 구현/설계에 직접 도움이 되는 결론을 우선한다.
- 모든 assistant 응답은 예외 없이 본문 시작과 본문 종료 지점에 동일한 배너 라인을 반드시 사용한다.
- 배너 형식은 다음과 같다.

`█████████████━━━━━━━━━━━━━━━━[제목]━━━━━━━━━━━━━━━━█████████████`

- `[제목]`은 이번 응답의 주제나 초점을 간단히 보여주는 짧은 제목이면 된다.
- 단순한 분류어보다, 응답이 다룰 내용이 드러나는 표현으로 작성한다.
- 예: `빈 등록 흐름 분석`, `트랜잭션 전파 규칙 설명`, `DispatcherServlet 구조 리뷰`, `AGENTS.md 문구 수정`
- 하나의 응답에는 위 배너가 반드시 2번 포함되어야 하며, 각각 응답의 시작과 끝에 위치해야 한다.
- 본문은 두 배너 사이에만 작성한다.
- 코드블록 내부에는 이 배너 규칙을 적용하지 않는다.


## 필요 시 강조할 관점
- 이 기술이 해결하는 실무 문제는 무엇인가?
- Spring은 어떤 추상화와 컴포넌트 경계로 해결하는가?
- 어떤 디자인 패턴/아키텍처 원리가 핵심인가?
- 확장 포인트와 커스터마이징 경계는 어디인가?
- 운영 관점(장애 대응, 디버깅, 관측성)에서 장단점은 무엇인가?
- Winter Framework에 옮길 때 최소 구현 경로와 리스크는 무엇인가?

## 선택적 추가 항목
- 관련 학습 포인트가 있으면 optional하게 함께 제시한다.
- 관련 키워드도 optional하게 덧붙인다.
- 키워드는 과하지 않게, 실제 다음 학습/구현에 도움이 되는 수준으로만 제시한다.

## 코드 작업 원칙
- 코드 작업 중 새로 추가하는 주석은 한글로 작성한다.
- 주석은 간결하고 명확하되, 처음 읽는 사람도 역할과 의도를 이해할 수 있을 정도로 작성한다.
- 주석은 현재 코드의 책임, 계약, 동작 기준을 설명하고, 미래 계획이나 추측은 적지 않는다.
- 클래스 주석은 이 타입의 역할, 사용 위치, 해결하려는 문제를 설명한다.
- 메서드 주석은 책임, 입력 기준, 핵심 분기, 반환 또는 실패 조건을 설명한다.
- 필드와 어노테이션 속성 주석은 값의 의미와 해석 기준을 설명한다.
- 주석은 "어떻게 구현했는지"보다 "왜 필요한지"와 "무엇을 보장하는지"를 우선해서 설명한다.
- 주석은 없는 차이를 만들지 말고, 실제 차이가 있다면 그 차이만 그대로 적는다.
- 현재는 테스트나 실행 기반 검증 환경이 없으므로, 변경 후 테스트 실행이나 애플리케이션 실행으로 검증하지 않는다.

## Code Response Rule
- 응답은 최대한 코드 기반으로 설명한다.

`[CodeSource] <path> Line <start>-<end>`

```<language>
<원문 코드 전체>
```

규칙:
- 전체 관점: 소스코드 제시 전에 이번 구간에서 무엇을 이해해야 하는지와 왜 중요한지를 먼저 제시한다.
- 구간 포인트: 이어서 나올 클래스/메서드별 역할, 호출 위치, 집중 포인트를 먼저 안내한다.
- 흐름 기준: 메서드 설명은 `진입 조건 -> 핵심 분기 -> 종료/반환` 순서로 정리한다.
- 각 `CodeSource` 코드블록 바로 위에 해당 메서드의 역할과 집중 포인트를 1~2줄로 먼저 제시한다.
- 역할/집중포인트 하이라이팅은 `diff` 블록의 `-` 라인으로 표시한다.
- 설명 요청에서는 기본적으로 원문 코드의 의미와 구조를 유지한 상태에서 `// [Codex]` 설명 주석이 포함된 단일 코드블록을 제공한다.
- 사용자가 원문만 요청한 경우에만 설명 주석 없는 순수 원문 코드블록을 제공한다.
- 원문 코드의 의미, 구조, 순서, 실제 코드 라인은 절대로 변경하지 않는다.
- 원문 코드의 실제 코드 라인은 절대로 생략, 요약, 재배치, 삭제, 치환하지 않는다.
- 설명을 위한 `// [Codex]` 주석 추가는 허용하며, 이는 원문 코드 훼손으로 보지 않는다.
- `// [Codex]` 주석을 추가하더라도 원문 코드의 의미, 구조, 순서, 실제 코드 라인은 절대로 바꾸지 않는다.
- 핵심 흐름이나 집중 포인트를 강조할 때는 코드블록을 끊은 뒤 별도의 `diff` 블록으로 하이라이트한다.
- 핵심 흐름/주석 하이라이팅은 `diff` 블록의 `+` 라인을 시각적 강조 용도로 사용한다.
- `diff` 블록의 `+` 라인은 조건문, 반환문, 대입문, 핵심 호출문처럼 실제로 집중해서 봐야 하는 코드 라인만 강조한다.
- `diff` 블록의 `+` 라인으로 코드 블록 전체를 과하게 감싸지 않는다.
- 하이라이트가 끝나면 남은 코드를 이어서 보여주기 위해 코드블록을 다시 연다.
- 하이라이트된 코드 라인은 다음 코드블록에서 즉시 반복하지 않는다.
- 코드 제시는 `강조 전 코드 -> diff 하이라이트 -> 강조 후 코드` 순서로 자연스럽게 이어지도록 분할한다.
- 코드블록을 다시 열 때는 직전 하이라이트 구간의 다음 줄부터 이어서 작성한다.
- `diff` 블록은 `-` 라인을 역할/집중포인트 하이라이팅에만 사용한다.
- 주석을 추가할 때는 반드시 `// [Codex]` 형태로 시작한다.
- Codex 주석은 코드 라인 맥락 안에서만 추가한다.
- `diff` 블록은 시각적 강조용이며 실제 코드 수정/추가/삭제 의미가 아니다.
- 코드 응답은 목적과 흐름이 이해되도록 필요한 맥락을 포함해 제시하고, 핵심 설명을 함께 제공한다.
- `CodeSource`는 항상 한 줄로만 작성한다.
- 경로 + `Line` 정보를 반드시 포함한다.
- 원문 코드 라인의 생략(`...`), 요약, 임의 수정은 금지하며, 추가 설명은 `diff +` 하이라이팅으로 보강하고 역할/집중포인트 강조는 `diff -`로 표기한다.
